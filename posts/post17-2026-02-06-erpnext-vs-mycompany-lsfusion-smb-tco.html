ERPNext vs MyCompany (lsFusion) for SMB: Unique processes, code, and the real cost of change

Both systems are free to start. The real difference for SMB is how much it costs to evolve unique processes over 2 years — with concrete numbers and code.

Who this article is for

  • SMB owners and CTOs
  • technical directors
  • business systems architects
  • anyone choosing ERP not for “right now”, but for the next 2–3 years

1. Key premise: both systems are free to start

Let’s start with what’s often missed.

  • ERPNext — open source, free.
  • MyCompany (lsFusion) — open source, free.

In both cases:

  • no license fees
  • no per-user pricing
  • an SMB can deploy the system, start using it,
  • and only then decide what to customize

So the question is not “what’s cheaper to start”.
The question is what happens next.

2. Why unique processes matter for SMB

In small and mid-sized businesses, processes are almost always:

  • non-standard
  • changing frequently
  • full of deliberate “rule exceptions”

Typical reality:

  • pricing is situational
  • reservation can be soft
  • margin can be negative — intentionally
  • managers must see consequences, not just get blocked

If ERP doesn’t support this natively, the business either breaks the system — or breaks itself.

3. A realistic baseline process (same for both)

Process:

Order → Pricing → Reserve → Ship

Conditions (what makes the business unique):

  • Price depends on:
    • customer
    • urgency
    • current stock
  • Reservation:
    • soft (can be overridden / reallocated)
  • Manager:
    • sees margin and risk
    • can confirm an “at-risk order”

This is not exotic — it’s normal SMB reality.

4. ERPNext: how this process is implemented

4.1 Architecture model

ERPNext is built around:

  • documents (DocType)
  • document lifecycle
  • event handlers (validate, on_submit, …)

Uniqueness = code around documents.

4.2 Pricing logic (code example)

# Server Script / Hook: Sales Order validate

def validate(doc, method):
    for it in doc.items:
        rate = it.base_rate

        if doc.is_urgent:
            rate *= 1.2

        stock = get_stock(it.item_code, it.warehouse)
        if stock < it.qty:
            rate *= 1.15

        it.rate = rate

4.3 Margin warnings

  • computed in Python
  • surfaced via frappe.msgprint() or JS form scripts
if it.rate - it.cost_rate < 0:
    frappe.msgprint(
        f"Negative margin for item {it.item_code}",
        alert=True
    )

4.4 Soft reservation

ERPNext does not have a first-class concept of “soft reservation”.

Common implementation:

  • a custom DocType Stock Reservation
  • extra checks in hooks
  • “allow if possible” logic

4.5 How many changes does this cost?

Over 2 years (real SMB):

8 business-logic changes:

  • new pricing formula
  • new reservation type
  • VIP exceptions
  • new warnings

Average one change in ERPNext:

  • Python: 12–16 h
  • JS form: 6–8 h
  • testing: 8–12 h

≈ 44 hours per change

5. MyCompany (lsFusion): same process

5.1 Architecture idea

lsFusion is a model, not event glue.

You describe:

  • what price is
  • what risk is
  • what reservation is

not:

  • where to catch an event

5.2 Price as a property (code)

price(order, line) =
    basePrice(line)
    * urgencyFactor(order)
    * stockFactor(line);

5.3 Margin and risk are properties too

margin(order, line) = price(order,line) - cost(line);
risk(order, line) = margin(order,line) < 0;

5.4 Soft reservation is part of the model

reservedQty(order, item) =
    IF softReserve(order)
    THEN requestedQty(order, item)
    ELSE 0;

5.5 UI follows the model

The form naturally:

  • shows margin
  • highlights risk
  • offers a “Confirm with risk” action
ACTION confirmWithRisk(order) {
    IF EXISTS line WHERE risk(order, line) THEN
        MESSAGE 'Order confirmed with risk';
}

5.6 How many changes does this cost?

One change typically includes:

  • property edits: 2–4 h
  • form updates (if needed): 2–4 h
  • testing: 4–6 h

≈ 18 hours per change
